시스템 아키텍처 관리 범위 정의서

개요

본 문서는 AI 기반 면접 플랫폼(A4 프로젝트)의 시스템 아키텍처 관리 범위를 정의하기 위한 문서이다.
시스템 아키텍처는 단순한 서버 구성도가 아니라, 서비스 전체가 안정적으로 동작하기 위한 기술적 골격과 구조적 경계를 정의하는 역할을 수행한다.

시스템 아키텍처는 인프라, 애플리케이션 구조, 데이터 구조, AI 모델 연결 구조, 보안 구조, 운영 및 확장 전략까지 포함하며, 세부 구현이나 코드 작성 수준까지는 포함하지 않는다.

시스템 아키텍처의 관리 범위

시스템 아키텍처는 다음 여섯 가지 주요 영역을 관리한다.

2.1 인프라 아키텍처 (Infrastructure Layer)

인프라 레벨은 시스템이 물리적·논리적으로 어디에서, 어떻게 실행되는지를 정의하는 영역이다.
해당 영역은 반드시 시스템 아키텍처 관리 범위에 포함된다.

관리 항목은 다음과 같다.

서버 구성 방식 (On-premise / Cloud)

GPU 및 CPU 분리 구조

Docker 및 컨테이너 구성 전략

네트워크 구조

포트 설계 및 방화벽 정책

배포 방식

스케일링 전략

장애 대응 전략

A4 프로젝트 기준 예시 구성 요소는 다음과 같다.

PostgreSQL + pgvector

Redis

FastAPI Backend

Celery Worker (GPU/CPU 분리)

Ollama 또는 llama-cpp 기반 LLM 서버

React-Vite Frontend

LiveKit(WebRTC) 기반 미디어 서버

위 구성 전체는 시스템 아키텍처 책임 범위에 해당한다.

2.2 애플리케이션 아키텍처 (Application Layer)

애플리케이션 구조는 시스템 내부의 논리적 구성과 모듈 간 흐름을 정의하는 영역이다.

관리 항목은 다음과 같다.

Backend API 구조 설계

서비스 레이어 분리 기준

모듈 구조 정의

인증 및 인가 구조

상태(State) 관리 방식

LLM 호출 흐름 정의

RAG 파이프라인 설계

비동기 처리 구조 설계

세션 흐름 정의

A4 프로젝트 기준 예시는 다음과 같다.

/interview/start
/interview/evaluate
/report/generate

또한 다음과 같은 전체 흐름 설계도 아키텍처 영역에 포함된다.

질문 생성 → 답변 수집 → 평가 수행 → 점수 산출 → DB 저장 → 보고서 생성

이러한 서비스 전체 처리 흐름의 정의는 시스템 아키텍처 책임 범위에 해당한다.

2.3 데이터 아키텍처 (Data Layer)

데이터 아키텍처는 데이터 구조 및 저장 전략을 정의하는 영역이다.
단순 테이블 생성이 아니라, 데이터 저장 기준과 구조 원칙을 정의하는 것이 목적이다.

관리 항목은 다음과 같다.

ERD 정의

테이블 설계 기준

인덱스 전략

VectorDB 구조 설계

대화 이력 저장 방식

세션 상태(State) 저장 전략

로그 저장 정책

트랜잭션 처리 전략

예를 들어, 면접 세션의 상태를 DB에 저장할 것인지 여부는 단순 구현 문제가 아니라 시스템 설계 결정사항이며, 이는 시스템 아키텍처 영역에 포함된다.

2.4 AI 모델 아키텍처

AI 모델 아키텍처는 모델의 세부 학습이 아니라, 시스템 내에서 어떤 모델을 어디에 배치하고 어떻게 연결하는지를 정의하는 영역이다.

관리 항목은 다음과 같다.

STT 모델 선정 및 연결 구조

LLM 모델 선정 및 호출 구조

평가 모델 처리 방식

Vision 모델 연결 구조

GPU 사용 전략

모델 간 호출 순서 정의

A4 프로젝트 예시는 다음과 같다.

STT → Whisper 계열 모델
LLM → Exaone 7.8B
평가 모델 → 프롬프트 기반 평가 구조
Vision → MediaPipe

다만 다음 항목은 시스템 아키텍처 범위에 포함되지 않는다.

모델 하이퍼파라미터 튜닝

데이터 증강

파인튜닝 세부 실험

모델 학습 데이터 제작

해당 영역은 모델 엔지니어링 범위에 해당한다.

2.5 보안 아키텍처 (Security Architecture)

보안 구조는 시스템 신뢰성과 직결되는 설계 영역으로, 아키텍처 관리 범위에 포함된다.

관리 항목은 다음과 같다.

인증(Authentication) 구조 정의

인가(Authorization) 정책 설계

역할 기반 접근 제어(RBAC) 전략

네트워크 접근 통제 전략

데이터 암호화 정책 (전송/저장 구간)

로그 감사(Audit) 정책

관리자 권한 분리 전략

보안은 구현 세부가 아니라 구조 설계 차원의 결정사항이다.

2.6 운영 및 확장 아키텍처 (Operational & Scalability Architecture)

시스템의 안정적 운영과 확장 가능성을 보장하기 위한 구조 정의 영역이다.

관리 항목은 다음과 같다.

모니터링 전략 정의

로그 수집 구조 설계

장애 복구 전략 정의

백업 및 복원 정책

버전 관리 전략

확장 시나리오 정의 (Worker 확장, LLM 교체 등)

성능 병목 대응 전략

GPU Worker 분리 구조, 비동기 처리 구조, LLM 처리 병목 대응 전략 등은 운영 아키텍처 범위에 포함된다.

시스템 아키텍처에 포함되지 않는 영역

다음 항목은 시스템 아키텍처의 직접 관리 범위에 포함되지 않는다.

UI 디자인 세부 구성

CSS 스타일 정의

프롬프트 문장 단위 설계

개별 SQL 쿼리 최적화

세부 코드 구현

모델 학습 및 실험 관리

시스템 아키텍처와 구현 영역의 구분

구조적 의사결정과 구현 작업은 명확히 구분되어야 한다.

구분 기준은 다음과 같다.

시스템 전체 구조에 영향을 주는 결정은 아키텍처 영역이다.
단순 기능 구현이나 코드 작성은 개발 영역이다.

예시는 다음과 같다.

ERD 정의: 시스템 아키텍처

테이블 생성: 개발 구현

API 설계: 시스템 아키텍처

API 구현: 개발 구현

모델 선정: 시스템 아키텍처

모델 튜닝: 모델 엔지니어링

결론

시스템 아키텍처는 단순한 기술 다이어그램이 아니라, 서비스가 안정적으로 운영되기 위한 전체 구조와 기술적 경계를 정의하는 역할을 수행한다.

따라서 시스템 아키텍처의 범위는 다음 한 문장으로 정의할 수 있다.

시스템 아키텍처는 서비스가 안정적으로 동작하기 위한 전체 구조와 기술적 경계 정의까지 관리한다.
