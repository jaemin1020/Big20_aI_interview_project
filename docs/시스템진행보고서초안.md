## 1. 프로젝트 총괄 개요 및 비전 (Project Overview & Vision)

### 1.1 프로젝트의 탄생 배경과 비전

Big20 AI 면접 프로젝트(A4 프로젝트)는 단순히 일방향적인 온라인 테스트 도구를 넘어서, 최전선 기술인 대형 언어 모델(LLM), 시각 인공지능(Vision AI), 음성 인식 및 합성(STT/TTS), 그리고 초저지연 양방향 통신망(WebRTC)을 한데 엮어낸 **실전형 차세대 면접 플랫폼**임.
본 시스템은 대형 클라우드 사업자의 매니지드 서비스(SaaS)에 종속되지 않고, 온프레미스(On-premise) 형태의 통제된 서버 네트워크 환경에서도 독립적이고 완벽한 B2B/B2C 면접 솔루션을 구동할 수 있도록 설계됨.

지원자의 답변 내용 자체의 논리성과 직무 적합성을 파악함은 물론, 발화 속도, 목소리 떨림, 시선 분산도, 미세 표정 변화 등 비언어적(Non-verbal) 태도까지 종합적으로 수집·분석하여, 사람 면접관보다 더 정량적이고 딥(deep)한 인사이트를 제공하는 종합 피드백 대시보드를 구축하는 것이 근본적인 비전임.

### 1.2 시스템 아키텍처 담당의 전략적 책무 및 전면적 라이브러리 스택 명세

광범위한 이기종 기술 스택이 동원되는 만큼, 시스템 아키텍처 부서는 단순 배포 스크립트 작성을 넘어 다음과 같은 전방위적이고 거시적인 기술 경계를 총괄하며, 이 모든 과정을 **Docker 기반 가상화 환경(Containerization)** 으로 완벽히 묶어냄. 하위 모듈별 요구되는 상세 프레임워크와 의존성은 다음과 같음.

1. **도커(Docker) 기반 인프라스트럭처 엔지니어링 (Infrastructure)**:

   - **도커(Docker) 전면 도입 사유 (Why Docker?)**: 
     - **의존성 지옥(Dependency Hell) 차단**: React(Node.js), FastAPI(Python 3.10), 미디어 서버(OpenCV, FFmpeg C++ 바인딩), GPU 기반 AI 워커(CUDA 12.x, PyTorch) 등 버전에 극도로 민감한 이기종 스택들이 호스트 OS 환경을 오염시키고 충돌하는 현상을 원천 방지함.
     - **환경 불일치 제로화(Zero Environment Discrepancy)**: "내 PC에서는 되는데 서버에서는 안 돼"라는 고질적 병폐를 끊어내고, 언제 어디서든(로컬 개발망, AWS, 사내 온프레미스 서버) `docker-compose up` 명렁어 한 줄로 100% 동일한 컨테이너 생태계가 부스팅 되도록 설계함.
     - **마이크로서비스(MSA) 스케일링**: 트래픽 폭주시 비즈니스 로직(FastAPI)이나 AI 추론 노드(Celery Worker)만 선택적으로 N개씩 복제 증설(Scale-Out)할 수 있는 독립된 런타임 캡슐을 제공받기 위함.
   - GPU 가속이 필수적인 컨테이너(LLM 추론 등)와 멀티코어 연산이 중심이 되는 컨테이너(파싱, WebRTC 스트리밍) 간의 자원 경합 방지를 위한 물리적 토폴로지 격리 설계.
   - 단일 `docker-compose.yml` 기반의 통합 오케스트레이션을 통한 완전한 시스템 이식성 확보. (NVIDIA Container Toolkit 연동을 통한 `deploy.reservations.devices` 설정 강제)
   - 격리된 내부망(VNet) 설정과 호스트 머신과의 포트 바인딩 정책(API 8000, WebRTC 8080/UDP 50000 대역), 네트워크 병목 제어 전략 수립.
   - **주요 기반 인프라 스택**: `Docker Engine`, `Docker Compose v2`, `NVIDIA CUDA Toolkit 12.x`
2. **Frontend 레이어 (User Interface & Client)**:

   - CSR(Client-Side Rendering) 기반 SPA 환경으로 실시간 양방향 미디어 및 폴링 처리.
   - **핵심 라이브러리 스택 (`package.json`) 및 역할**:
     - `React (18.2.0)`: 가상 DOM을 이용한 컴포넌트 기반 UI 렌더링 및 Hooks 중심 생태계 관리.
     - `react-router-dom (6.21.0)`: 페이지 리로딩 없는 화면 전환 및 대시보드 컴포넌트 라우팅 제어.
     - `Vite (5.0.8)`: 기존 Webpack 대비 빌드 및 HMR(Hot Module Replacement) 속도가 압도적인 React 스캐폴딩.
     - `axios (1.6.2)`: Backend-Core의 REST API와 비동기 JSON 기반 통신 수행 및 헤더(JWT 토큰) 공통 제어.

     - `html2canvas (1.4.1)` & `jspdf (2.5.2)`: HTML DOM 트리를 스크린샷 캔버스로 변환하여 즉석에서 PDF 브리핑 리포트 다운로드 기능 보조.

## 2. 초기 아키텍처 설계안 구상 및 한계점 대면 (Initial Design & Limitations)

프로젝트 기획이 막 시작되었던 단계에서는, 최우선 목표가 "PoC(개념 증명)를 위한 빠른 프로토타이핑"에 맞춰져 있었음. 이에 따라 아키텍처는 극단적으로 단순화된 **모놀리식 특성을 지닌 제한적 분산 구조**를 채택했음.

### 2.1 초기 인프라 및 AI 처리 설계안

- **모놀리식 단일 컨테이너 및 볼륨 통짜 공유 성향**: Frontend(React), Backend(FastAPI), Database(PostgreSQL), Media-Server를 한 묶음으로 올리면서, 백엔드 컨테이너 자체가 가벼운 AI 추론을 같이 담당하거나 단 1개의 범용 AI Worker만이 전체 AI 연산을 떠맡도록 구성했음. 모든 정적 미디어(이력서, 음성)마저 하나의 `/app/data` 호스트 볼륨에 몰아넣어 호스트 레벨의 I/O 경합을 초래하는 형태였음.
- **파이썬 GIL(Global Interpreter Lock) 한계와 동기식(Synchronous) LLM 호출**: Frontend에서 유저 발화가 끝난 이벤트를 전송하면, 백엔드는 즉시 외부LLM 혹은 로컬LLM 엔진에 질문 텍스트 생성을 요청(Request)했음. 파이썬의 GIL 특성상 이 연산을 대기하는 동안 **FastAPI 쓰레드는 10~20초간 완전한 블로킹 대기(Blocking)** 에 빠졌으며, 이는 다중 유저 접속 시 전체 시스템의 Queue를 얼어붙게 만드는 전형적인 강결합 호출망임.
- **Memory-based 상태 관리 및 트랜잭션 부재**: 면접 세션의 진행 상태(현재 몇 번째 질문인지, 사용자의 누적 대화 내역이 무엇인지)를 백엔드 서버 RAM 변수(Dict)에 의존하여 임시 적재하고, 면접이 온전히 끝날 때만 DB에 덤프격으로 INSERT를 시도하는 방식임. 컨테이너가 1초라도 튕기면 해당 세션 데이터가 100% 휘발되는 영속성 부재의 약점을 지녔음.

### 2.2 한계 부딪힘 현상 (Bottlenecks & Death Spirals)

본격적으로 무거운 STT 기반 음성 인식이 추가되고, 브라우저가 캠 영상을 WebRTC(UDP)로 끊임없이 밀어 넣기 시작하자 빙산 아래 숨겨진 치명적인 구조적 결함들이 동시다발적으로 폭발했음.

1. **VRAM 자원 록(Resource Lock)과 모델 컨테이너 OOM 강제 종료**: 1개의 AI Worker 안에서 STT가 오디오를 텍스트로 변환(CPU+GPU 혼용)하는 동안, 동일 컨테이너 안의 LLM 추론 스레드가 남은 GPU 메모리를 순간적으로 고갈시킴. 이 경쟁 상황으로 인해 Linux 커널의 OOM(Out of Memory) Killer가 발동하여 프로세스가 무자비하게 강제 종료(Exit Code 137)되는 충돌이 잦았음.
2. **소켓 타임아웃(Socket Timeout)과 데드락 (504 Error)**: 앞선 동기식 구조 탓에, AI 모델이 일시적으로 연산이 밀려 30초 이상 HTTP 응답(Response)을 돌려주지 못하면 Nginx나 프론트 로컬 프록시 단말에서 Gateway Timeout(504)을 띄워 연결을 끊어버림. 이는 프론트 화면 전체가 하얗게 죽어버리는 클라이언트-서버 간 '데드락 아키텍처'였음.
3. **P2P 미디어 통신망 (WebRTC) 유실 현상**: 별도의 독립된 Media Server가 스트림을 전담 소화하지 못하고 Backend 코어 단에서 너무 큰 부하가 겹치다 보니 서버의 Event Loop 틱(Tick)이 밀리게 되고, 이로 인해 필수적인 RTP 패킷 수천 개가 드롭(Drop)되며 영상 싱크가 어긋나는 네트워크 병목이 발생했음.

이러한 문제들은 확장이 절대적으로 불가능한(Un-scalable) 구조임을 방증했으며, 즉각적이고 대대적인 아키텍처 진화(Evolution)의 계기가 되었음.

## 3. 컴포넌트별 아키텍처 초정밀 설계 진화 과정 (Evolutionary Details)

병목 현상들을 격파하기 위해, 컴포넌트 간 결합도(Coupling)를 낮추고 응집도(Cohesion)를 높이는 **MSA(Microservices Architecture) 지향적 리팩토링**이 전방위적으로 추진되었음.

### 3.1 인프라스트럭처 레이어의 고도화 (Infrastructure Isolation & Topology)

#### 3.1.1 GPU/CPU 이기종 자원 물리적 격리 (Workload Segregation)

전체 시스템의 생사를 쥐고 흔들던 AI 연산 충돌 사태를 진압하기 위해, 단일 워커 구조를 피기백(Piggy-back) 방식에서 벗어나 Celery Worker 컨테이너를 물리적, 논리적으로 두 개의 결계로 분할했음.

- **`interview_worker_gpu` (`gpu_queue` 전담 로드 밸런서)**:
  - `nvidia-container-toolkit` 데몬과 `docker-compose`의 `deploy.resources.reservations.devices` 블록을 결합하여 호스트의 NVIDIA VRAM 가속을 단독으로 독점.
  - 임베딩 변환기(HuggingFace), 면접 꼬리질문 생성기, 세션 최종 종합 채점기 등 무거운 행렬 연산 기반 텐서 추론(Tensor Inference) 태스크만을 구독(Subscribe)하도록 Celery 라우팅 룰을 하드코딩.
- **`interview_worker_cpu` (`cpu_queue` 전담 로드 밸런서)**:
  - 호스트 OS의 다중 코어 CPU 병렬 처리 능력을 극대화하기 위해 스레드 및 프로세스 풀 스케일링 적용.
  - 업로드된 수십 장짜리 이력서(PDF) 텍스트 및 JSON 정밀 구조화 파싱 작업, 오디오 메모리 버퍼 (`ndarray`) 변환, Faster-whisper 기반 STT 등 철저히 I/O 및 CPU 집약적 연산을 전담.
- **결과 (Impact)**: LLM 텐서 코어가 VRAM을 100% 긁어 쓰며 응답 텍스트를 직조(Weaving)하고 있더라도, CPU 워커는 완전히 독립된 프로세스 스페이스에서 멈춤 없이 새로운 음성을 텍스트로 트랜스포밍하는 **"완벽한 무간섭 비동기 평행망(Interference-Free Parallel Network)"**을 구축함.

#### 3.1.2 WebRTC 네트워크 토폴로지 교정 및 ICE 홀펀칭 유도

- 화상 면접의 심장인 웹 브라우저와 `aiortc` 미디어 서버 간의 통신은 TCP가 아닌 UDP 기반의 ICE(Interactive Connectivity Establishment) 교섭을 생명선으로 요구함.
- 단일 Docker Bridge 네트워크 내부에 갇힌 컨테이너 구조의 한계로 Candidate Gathering(공인/사설 IP 수집) 단계가 실패하는 현상(블랙스크린)을 타파하기 위해, 호스트 OS와 컨테이너 간에 트래픽 릴레이를 허용하는 **Port Binding Topology**를 재설계함.
- `docker-compose.yml` 상에 P2P 통신용 미디어 패킷 포트 대역(예: `50000-50050/udp`)을 명시하여 호스트OS에 매핑, WebRTC RTP 패킷이 도커 NAT를 통과하게 열어주는 터널링을 개통했음. 이를 통해 TURN 서버 없이도 로컬 및 폐쇄망에서 딜레이 0.5초 이내의 초저지연(Ultra-low Latency) 양방향 스트리밍 도로를 완공함.

#### 3.1.3 무중단 상태 자동화 DB Init 및 시딩(Seeding) 체계

- PostgreSQL이 올라오는 즉시, 애플리케이션과는 독립된 인프라 레벨의 Init 스크립트가 돌아가도록 컨테이너 `depends_on` 체인을 구성했음.
- 시스템 부트스트랩 단계에서 SQLModel `db_models` 팩토리를 스캔하여 미생성된 스키마 테이블을 안전하게 Create(멱등성 보장)하고, 시스템에 필수적인 마스터 관리자(`admin`) 계정과 IT/경영/기획 직군별 표준 회사 룰셋(Rule Set) 레코드를 자동 주입(Seeding)함. 컨테이너를 언제든 부수고 다시 빌드(Down->Up)해도 서비스 운영에 필요한 뼈대 데이터가 순간 복구되도록 인프라 회복 탄력성(Resilience)을 영구 보장함.

### 3.2 애플리케이션 및 비동기 이벤트 레이어 (Async Event-Driven Core)

#### 3.2.1 동기적 블로킹 파괴와 Redis 기반 비동기 RPC (Remote Procedure Call)

클라이언트의 HTTP 요청 대기 상태에 따른 서버 사망 문제를 근절하기 위해, 백엔드의 동작 원리를 철저히 Event-Driven 메커니즘으로 재설계함.

- 클라이언트(React)가 "다음 질문 주세요"라고 API(`GET /next_question`)를 호출할 때, 백엔드(FastAPI)는 직접 파이썬 내부 함수로 LLM을 동기 호출하지 않음.
- 대신, 즉각적으로 `celery_app.send_task("task_generate_question")` 매직 스트링 메서드를 통해 함수 명세와 파라미터를 JSON 직렬화(Serialization)하여 `Redis` 브로커의 대기 큐로 Push함. 이후 FastAPI는 단 0.05초 만에 클라이언트에게 "Accepted(HTTP 202)"를 리턴하고 다른 유저의 Request를 받을 준비함. (완전한 Non-Blocking I/O 스위칭 완료).
- 프론트엔드는 블로킹 되지 않은 쾌적한 화면을 유지하며 주기적으로 Rest API를 폴링(Polling)하여 면접 질의내용과 추출된 음성 파일(TTS) 갱신 결과를 비동기적으로(Asynchronously) 동기화받아오는 진일보한 프론트-백 연동 패턴을 안착시킴.

#### 3.2.2 무상태성(Stateless) 아키텍처와 분산 메모리 인프라

과거 메모리가 통째로 크래시를 일으킬 때 면접 진행률도 날아갔던 구시대적 Fault 방식에서 벗어나,
세션의 거시적 맥락(현재 1번 문항 진행중인가, 유저가 방금 한 말이 무엇인가 등)을 전부 PostgreSQL 트랜잭션으로 기록(`status: LIVE`, `transcript` 테이블 등)하여 서버 데몬 노드가 몇 번 죽든 영속성을 유지하도록 앵커링함.
반면 트랜잭션 오버헤드를 줄이기 위해, JWT 토큰 리프레쉬, 일회성 인증 OTP, WebRTC Signalling ICE 커넥션 객체 같이 초당 수 십 번 참조되는 단발성(Ephemeral) 데이터들은 `Redis` 의 In-Memory Key-Value 풀에 TTL(Time To Live)을 걸어 저장함으로써 Scale-Out 환경에서도 모든 API 노드가 동일 상태값을 참조하도록 메모리 한계를 우회함.

### 3.3 데이터 아키텍처 및 융합 검색엔진망 (PostgreSQL RAG)

#### 3.3.1 RDBMS와 Vector 서치의 원자 단위 융합: `pgvector` 의 이식

본 시스템은 "지원자의 이력서", "기업의 인재상", "기존 출제되었던 꼬리질문" 세 가지 차원을 복합 참조(RAG)하여 궁극의 맞춤형 프롬프트를 엮어냄.
이때 기존에는 RDBMS 인스턴스와 더불어 고수준의 분산 Vector DB(Milvus, Pinecone 등)를 이중으로 운영해야 하는 파편화가 존재했으나, 시스템 복잡도를 극소화하면서 단일 연결점(Single Point of Truth)을 확보하기 위해 PostgreSQL 18 인스턴스의 C 커널 레벨에 `pgvector` 확장을 탑재함.

- **RAG 임베딩 연산**: 지원자가 이력서 PDF를 올리면 -> CPU 워커가 텍스트를 논리적 문단으로 분할(Chunking)하고 -> GPU 워커가 이를 HuggingFace 로컬 모델(Sentence-Transformer)을 통과시켜 정확히 `1024차원`의 고밀도 부동소수점 임베딩 벡터 배열 `[0.012, -0.344, ...]` 로 압축함.
- **수학적 Hybrid Querying (SQL + 거리연산)**: 압축된 벡터를 DB의 `resumes.embedding (vector(1024))` 컬럼으로 다이렉트 저장함. 면접관(LLM)이 질문을 구성할 때, 단순 SQL문장의 `WHERE` 절에 FK 조건을 걸고, `ORDER BY resume.embedding <-> 질문벡터` 구문(L2 Distance)이나 `<=>` 구문(Cosine Similarity)을 날려 "RDBMS의 접근 권한 제어"와 "AI 의미론적 유사성 탐색"을 단 한 번의 Query 조인으로 반환받는 압도적 아키텍처 효율을 달성함.

#### 3.3.2 데이터 정규화 및 ERD 인덱스 고도화

- 무한 확장이 가능해야 하는 트랜스포머 구조에 맞춰 Backend의 `SQLModel`(SQLAlchemy 기반) 코드를 수차례 리팩토링했음. 특히 가장 Insert가 빈번한 `Transcripts`(대화 기록) 로우가 어떠한 면접(`interview_id`)의 어느 질문(`question_id`)의 답변인지를 명확히 외래키(FK)로 트래킹할 수 있도록 연관 매핑 무결성을 단단히 조율함.
- `Emotions` 같이 초당 5개씩 쏟아지는 감정 분석 지표들은 별도 테이블로 분할(EAV/Anti-pattern)하여 엄청난 Join 과부하를 일으키지 않고, Transcripts 테이블 보디 내부의 `JSONB` 멀티컬럼(`emotion_summary` 등)으로 원자화 편입시켰음. Postgres의 `GIN(Generalized Inverted Index)` 인덱스 지원을 통해 I/O 레이턴시 증가 없이 막대한 비정형 통계 데이터를 거뜬히 소화하고 있음.

### 3.4 AI 모델 최적 서빙 및 미디어 파이프라인 (AI Serving & RTSP)

#### 3.4.1 Client-Side/Server-Side 하이브리드 지향적 STT 오디오 파이프라인

구조상 미디어 서버단에서 영상을 파이프하는 중간에 FFmpeg등으로 오디오 프레임 추출을 병행하여 서버 로컬 모델로 넘기는 초기 구상은, 도커 내부 소켓 연결 폭파 및 GIL 데드락을 유발하는 생태계의 원흉임. 이를 **Client-side Audio Buffering Architecture**로 역발상 전환함.

- 프론트(React 브라우저)단의 Web API인 `AudioWorkletNode`를 호출하여 백그라운드 워커 스레드로 마이크 PCM 고음질 튜닝 버퍼 배열(`Float32Array`)을 직접 분리 수집함. 이를 통해 브라우저 메인 가상 DOM UI 렌더링 스레드는 프레임 드랍(Frame-Drop) 없이 절대 보호를 받음.
- 유저 화상 패킷(UDP)은 WebRTC를 타고 Media-Server로 직행하지만, 끊임없이 발화되는 음성 청크 덩어리 자체는 브라우저 단에서 고속 API 결합을 이용해 백엔드의 오프라인 Whisper 엔진으로 병렬 발사됩음. 이 분산 아키텍처를 통해 유저가 말하는 즉시 STT 처리가 백그라운드 `cpu_queue` 워커로 격리 위임되어 안정적인 리얼타임 레이턴시를 쟁취함.

#### 3.4.2 Vision 연산과 백그라운드 Time-Delta 샘플링 방어막

- AI Media-Server에서는 들어오는 비디오 패킷에서 `RTP` 프레임 헤더를 뜯어내고 파이썬 `ndarray`로 배열 조각을 훔쳐(Overriding Protocol), 비전 백본넷인 MediaPipe Face Landmark 엔진에 공급함.
- 그러나 초당 30번(30 FPS) 쏟아지는 전체 프레임을 무식하게 모델에 밀어 넣으면 CPU 코어 점유율이 즉시 100% 한계치에 다다라 Host Machine 전체 크래시(Thermal Throttling)가 발생함.
- 아키텍트는 이를 방어하기 위해 프레임 수신 이벤트 콜백(Callback) 단에 파이썬의 `time.time()` 기반 강제 타임 델타 로직을 적용, **"초당 최대 5 프레임만 스캔(Frame Skipping Sampling)"** 하도록 방어벽을 설계했음. 그 결과 서버의 컴퓨팅 부하를 80% 덜어내면서도 "발화 간 긍정 40%, 무표정 60%"라는 통계적 샘플링 신뢰구간은 완벽히 유지해내는 획기적 최적화를 입증함.

#### 3.4.3 스레드 안전성(Thread-Safety) 기반 오프라인 고성능 TTS 파이프라인

- 면접관(LLM)이 문장을 완성하자마자 곧바로 한국어 음성 피드백을 브라우저에 렌더링하기 위해, **Supertonic 2** 오프라인 오디오 모델을 `cpu_queue` 워커로 위임 마운트함.
- **CPU 경합 차단과 쓰레드 락(Lock) 컨트롤**: 무거운 오프라인 텐서 연산이 전체 서버 스레드 풀을 말라죽게(Deadlock) 하는 참사를 방지하기 위해 컨테이너 내부 연산 코어를 `torch.set_num_threads(1)`로 고립시켰으며, 요청이 폭발해도 파이썬 `threading.Lock()`을 묶어 반드시 한 명의 요청 처리만 안전하게 보장하도록 뮤텍스(Mutex) 방어선을 전개함.
- **도커 공유 볼륨 캐싱(Double Check Hashing)**: 한 번 LLM이 뽑은 질문 식별자(`q_{question_id}.wav`) 기반 캐시 파일을 도커 백엔드와 공유하는 Host Volume에 직접 내려적음. 작업 큐가 동시다발적으로 들어와도, 락을 열기 직전과 직후 파일 객체의 물리적 Size 무결성을 이중 체크(Double Check)하여 이미 생성된 음성이면 즉각 연산 스킵 코드를 발동시켜 스루풋 부하를 완전히 쳐내는 최적화 기조를 달성함.

## 4. 시스템 아키텍트의 독보적 개발 성과 및 기술적 기여 (Architect Contributions)

### 4.1. 베이스 인프라 및 모노레포(Monorepo) 생태계 정립

서로 다른 프레임워크(React, FastAPI, aiortc 등)가 난립하던 파편화된 구동 환경을 통합 통제하기 위해, 진입점을 단일화함.

- 최상위 루트 디렉토리를 기준으로 `frontend`, `backend-core`, `ai-worker`, `media-server`의 모노레포(Monorepo) 구조를 최초 생성하고 각 서비스 데몬의 진입점(Entrypoint)을 통일화함.
- 복잡한 네트워크 명령어 조합 없이 단순 `docker-compose up --build` 명령어 단 하나만으로, 4개의 이기종 네트워크 컨테이너와 2개의 DB 인스턴스가 상호 간 DNS(`http://interview_backend:8000` 형태)를 알아보고 약속된 포트 매핑 하에 일제히 기동 될 수 있도록 Docker Network Bridge 뼈대를 완전 구상 및 이식함.
- Git 트래킹 시 불필요하게 딸려 올라가는 GPU 가중치 파일(`*.gguf`, `*.safetensors`), 패키지 캐시(`__pycache__`, `node_modules`), 개인 API 키가 들어있는 `.env` 파일들을 루트 및 하위 모듈별 2중 `.gitignore` 필터링 규칙으로 섬세하게 직조해 형상 관리 리포지토리를 초경량화함.

### 4.2. 코어 비즈니스 지형도 파괴적 혁신 및 AI/데이터 결합

- **초기/에러 대비 이중 Fallback (안전장치) 시스템 구현**:
  - 시스템 파일 모듈이 로드되지 않거나 에러를 뿜으며 면접 세션 생성이 완전히 중단될 뻔한 위협들을 방어함.
  - 예를 들어 `interview_scenario` 설정 파일 로드에 실패하여 참사가 벌어질 법한 환경에서도 파이썬 `try-except ImportError` 블록의 흐름을 꺾어 **서버 코드 내 하드코딩된 사전 예비 문항(Fallback Default Questions) 배열을 강제로 로드**시키고 정상 면접 플로우로 탑승시키는 방어 트랜잭션 로직을 이식하여 시스템 유연성을 고취시킴.
- **RAG / Vector DB 스키마 100% 동기화 작업**:
  - 백엔드 레이어의 `db_models.py`와 `database.py` 안에 혼재되어 있던 복잡한 1:N 쿼리 연결, 그리고 LLM이 요구하는 `1024차원 vector` 컬럼 선언문을 손끝으로 하나하나 재검수했음. 데이터베이스 마이그레이션(Entity) 구조와 물리적 SQL 테이블이 완전한 무결성을 이루도록 튜닝을 실행함.

### 4.3. 대형 TDD / 기술 명세서(Technical Docs)

개발된 코드 조각들이 휘발되지 않도록, 전체 시스템의 작동원리와 구조를 파헤치는 대규모 기술 문서화(Technical Writing) 작업을 아키텍트 주관 지휘하에 프로젝트 처음부터 끝까지 수행함.

- **시스템 아키텍처 디자인 명세서(SAD) 전면 고도화 (v1.4.0)**:
  - 단순 서술 수준이던 문서를, C4 모델 정적 구조 분해(Frontend~AI Worker 계층별 통신 프로토콜 매핑), 런타임 동적 시스템 워크플로우(WebRTC 미디어 패킷 핸드셰이킹부터 LLM 추론까지 이어지는 완전 무결한 비동기 핑퐁 프로세스) 시퀀스 분해로 탈바꿈시켜 초정밀 머메이드(Mermaid) 다이어그램 명세로 배포 완료함.
- **문법 포대갈이 및 구버전 호환 렌더링 튜닝**:
  - `actor` 키워드 오류나 마크다운 파서 에러로 인해 구버전 환경(Mermaid 8.8 이하)에서 다이어그램이 깨지던 현상을 타파하고자, `participant` 로의 문법 컨버팅, 노드 괄호 디버깅, 다이어그램 재포맷팅 작업을 기를 쓰고 완수함.
  - 그 밖의 플랫폼 보안, DB 연동 가이드, 트러블슈팅 열전, 최적화 레거시 리포트 등 수십여 개의 마크다운 지식 베이스가 아키텍트의 손끝에서 편찬되어, 사실상 AI 면접 생태계 유지보수의 바이블(Bible) 인프라로 안착함.

### 4.4. 협업을 위한 깃(Git) 브랜칭 전략 수립 및 형상 관리 고도화

이기종 컴포넌트(AI 워커, Backend, Media 등) 파트별 병렬 개발 통제력을 확보하고 병합 충돌(Merge Conflict)을 체계적으로 방어하기 위해, 브랜칭 및 커밋 워크플로우를 아키텍처 수준에서 표준화함.

- **맞춤형 Git Flow 브랜칭 전략 정립**:
  - 언제든 프로덕션 배포가 가능한 안정된 베이스 캠프인 `main` 브랜치를 통제된 승인망 아래 보호함.
  - 매일의 통합 및 런타임 테스트를 수행하는 `develop` 브랜치를 중추로 삼고, 각 팀원들이 `feature/{이슈/기능명}`, `fix/{버그명}` 네이밍 규칙에 맞춰 로컬 작업을 격리하도록 룰을 강제함.
- **모노레포 특화 코드 리뷰 체계**:
  - 한 저장소에 프론트, 백, AI, 미디어가 모두 모여 있으므로(Monorepo), 개발자가 PR(Pull Request)을 날릴 때 사이드 이펙트를 최소화하기 위해 '수정된 디렉토리 도메인'을 명확히 태그로 선언하게 유도함.
- **인프라 변경 롤백(Roll-back) 안전망**:
  - 핵심 `db_models.py` 데이터 모델 변경이나 도커 환경 변수(`docker-compose.yml`) 수정 같이 파급력이 전 시스템에 미치는 인프라성 커밋은, 별도의 `chore/infra-` 브랜치로 분리하여 아키텍트의 직접 리뷰(Approve)를 필수로 거치도록 거버넌스를 이식함.

## 6. 치열한 기술 병목 극복 및 트러블슈팅 열전 (Critical Troubleshooting Portfolio)

실제 프로덕션 서버를 빌딩하면서 발생한 10가지 이상의 대형 참사와 이를 엔지니어링 솜씨로 무력화시킨 여정임. 문서 용량 제약상 핵심 사건만 상세 발췌함.

### 6.1 치명적 `ModuleNotFoundError`: Celery 네임스페이스 격리 실패 사태

- **증상 관찰**: Backend 컨테이너가 면접 꼬리 질문을 발행하기 위해 `@celery_app.task` 로 등록된 워커의 함수를 직접 `from tasks.llm_generator import generate_question` 문법으로 호출하는 순간, FastAPI 컨테이너가 Pydantic ValidationError 및 `ModuleNotFoundError`를 뿜으며 크래시가 터짐.
- **원인 기전 분해**: 도커 상 물리적으로 백엔드와 AI 워커는 완전히 "분리된 컨테이너 공간과 분리된 파일시스템"을 갖음. 따라서 물리 모듈을 파이썬 내장 `import`로 끌어오는 것은 분산 처리 MSA 사상에 정면 위배되는 아키텍처 결함(Tight-Coupling) 임.
- **설계적 처방**: "Signature" 패턴 기반 위임으로 아키텍처 리팩토링 수행. Backend 코어에서는 어떠한 워커 쪽 스크립트 도 가져오지 않도록 코드를 갈아엎고, `celery_app.send_task("generate_question", args=[session_id, transcript], queue="gpu_queue")` 매직 스트링 포맷을 도입했음. 오직 Redis 메시지 브로커 안에 JSON 직렬화 명세표만 밀어 넣고 응답은 버리는 **"완전한 객체 지향적 원격 프로시저 호출(RPC) 디커플링"**을 이룩함.
- **증명된 결과 (Result)**: 백엔드 모듈 충돌 에러율이 0%로 수렴함. LLM 워커 컨테이너가 아무리 재부팅되더라도 백엔드 웹서버(API)의 생존성이 완벽하게 보장되는 궁극의 장애 격리(Fault Isolation) 체계를 완성함.

### 6.2 오디오 메인 렌더 스레드 점유로 인한 브라우저 `Event Loop` 블로킹 현상

- **증상 관찰**: React 프론트엔드가 면접장에 들어가면 노트북 팬(Fan)이 굉음을 내면서, Canvas 애니메이션과 마우스 스크롤이 끊기는 치명적 렉(Lagging: 15 FPS 이하 하락) 현상이 유발됨.
- **원인 기전 분해**: Web API `getUserMedia` 로 얻어온 오디오 마이크 PCM 버퍼 스트림을 레거시 규격인 `ScriptProcessorNode`의 콜백(Callback) 이벤트로 넘겼기 때문임. 이 노드는 자바스크립트의 "단일 메인 렌더 스레드(Main UI Thread)" 에서 배열을 썰고 파싱하느라 V8 엔진의 Event Loop 틱을 전부 점유함.
- **설계적 처방**: 메인 스레드의 부하를 덜어내기 위해 브라우저 가상 머신(VM) 백그라운드로 도피하는 최신 모던 API **`AudioWorkletProcessor`** 구조판으로 웹 오디오 코어 엔진 모듈 마이그레이션 단행. 크로스 오리진(CORS) 대응 후 Worker 스레드로 연산을 완전히 분리함.
- **증명된 결과 (Result)**: 마이그레이션 적용 즉시 브라우저 메인스레드 점유율(CPU Time)이 기존 대비 60% 이상 극감함. 10분 연속 음성 캡처 상황에서도 프레임 드랍 없이 부드러운 60 FPS 렌더링 유지력을 증명하며 프론트 데드락 병목을 영구 퇴출함.

### 6.3 볼륨 바인딩(Bind Mounts) 권한 충돌과 이력서 파싱 증발 파동

- **증상 관찰**: 사용자가 PDF 이력서 파일을 올리고 Backend가 분명히 업로드 성공 응답(HTTP 200 OK)을 쏴줬는데, 30초 뒤 DB를 열어보면 이력서 JSON 파싱 상태가 통째로 `failed` 거나 `None` 으로 비어있었음. Worker 로그에는 `FileNotFoundError: [Errno 2] No such file or directory` 에러가 적혀있었음.
- **원인 기전 분해**: Backend 컨테이너 단의 `/app/uploads` 폴더 공간과 CPU Worker 컨테이너 파일 시스템 공간이 완전히 남남으로 독립되어 있어 파일이 Host 디스크를 거쳐 넘어가지 못하고 고립(Isolation)되었던 인프라 볼륨 결함임.
- **설계적 처방**: `docker-compose.yml` 리소스 명세단에 `- ./backend-core/uploads:/app/uploads` 와 같이 호스트 디렉토리를 바인드 마운트(Bind Mount)하여, 백엔드 컨테이너와 AI 워커 컨테이너가 동일한 물리적 호스트 파일 시스템 경로를 매핑하여 참조하도록 관통 연결 브릿지를 설정함으로써 이기종 컨테이너 간의 Zero-copy 파일 전달 IO 장애를 말끔히 해결함.
- **증명된 결과 (Result)**: 볼륨 바인드 처리 후 파일 증발(Data Loss) 에러율이 완전히 0%로 박멸됨. 사용자가 이력서를 던지는 즉시 워커 노드가 File I/O 블로킹 없이 원본 PDF를 스캔해갈 수 있는 무결성 파이프라인이 정립됨.

## 7. 시스템 스펙 및 성능 검증 벤치마킹 분석 리포트 (Performance & Benchmark)

자원 최적화 설계의 과실을 숫자로 대변하는 핵심 퍼포먼스 체크포인트 요약.

- **미디어 스트리밍 실시간성(RTSP) 및 네트워크 지터(Jitter) 방어**:
  - Host OS 포트바인딩 기반 UDP 홀펀칭 트래버스 적용 전에는 프레임 드랍률이 15% 이상 치솟았으나, 포트 개방 반영 후 네트워크 지터(Jitter)로 영상/음성이 심하게 밀리던 현상 100% 제압.
  - 평균 화상 전송 RTT(Round Trip Time) 왕복 **150ms ~ 300ms**의 극도의 초저지연 레이턴시 풀 달성으로 기계와 대화하는 듯한 불쾌한 핑퐁 장애 제거.
- **대규모 추론 (EXAONE-3.0-7.8B-Instruct) 병렬 소요시간**:
  - Task 큐잉 딜레이 및 GPU(VRAM) 텐서 로딩, 그리고 RAG 유사도 검색 연산 비용까지 통합 스코어링.
  - 지원자의 답변 텍스트(STT 결괏값)가 꽂힌 순간부터 RAG 컨텍스트를 조립해 완전히 새로운 프롬프트를 만들고, 날카로운 꼬리질문 텍스트가 생성되어 DB에 저장된 후 프론트엔드 폴링(Polling)을 통해 화면에 떨어질 때까지의 트랜잭션 엔드-투-엔드(End-To-End) 평균 생성 리드타임을 **7.5초 ~ 12초** 내외 안전권으로 묶어냄. (Gateway Timeout 0건 발생 관측 완료)
- **STT (Faster-Whisper) 청크 처리 스루풋 (Throughput)**:
  - 클라이언트에서 보내오는 5초/10초 단위의 정적 감지(VAD) 오디오 웨이브 덩어리(`.wav` 버퍼 블록)들을 텍스트로 트랜스포밍하는데 걸리는 모델 추론 소요 시간 단 **1.2초 ~ 1.8초**.
  - 화자가 말을 멈춤(Silence Detection)과 동시에 프론트 측에서 패킷을 쏘고, CPU Worker가 낚아채 오프라인 로컬 엔진(Faster-Whisper)으로 변환하는 수준 높은 멀티 프로세스 스루풋 입증. CPU 100% 코어 점유율 피크 시에도 다른 연산이나 Media, Backend API 컨테이너에 영향을 일절 주지 않음.

## 9. 프레임워크 수준의 보안 하드닝 및 데이터 컴플라이언스 쉴드 (Security Stance)

- **JWT 기반 Stateless 토큰 암호화 체계 (PyJWT & Passlib)**: 서버 메모리에 유저의 로그인 세션 상태를 전혀 남기지 않는 (Redis I/O조차 아끼는) 완전한 RESTful 원형 유지. 시크릿 대칭키 해시 알고리즘(`HS256`) 기반으로 토큰 위변조를 원천 봉쇄. `passlib[bcrypt]`를 통한 `Work Factor 12` 레벨 비밀번호 난독화 스키마 통합 및 FastAPI `OAuth2PasswordBearer` Depends 체인 인증망 완비함.
- **엔드포인트별 치밀한 권한 분리 통제방 (RBAC - Role Based Access Control)**:
  - `admin`, `recruiter`, `candidate` 3계층 유저 팩션을 정의하고 DB Role Enum 매핑.
  - 본인 이력서 및 트랜스크립트 파싱 결과를 몰래 남의 계정으로 객체 식별자(ID)를 변조 유추하여 통과하려는 IDOR (Insecure Direct Object Reference) 해킹 공격을 도메인 모델 수준에서 `if current_user.id != request.user_id` 방어 트리거로 튕겨버려(HTTP 403 Forbidden) 엄밀한 멀티 테넌트 무결성을 수호함.
- **클라우드 네이티브 운영 체제 환경 은폐화 (.env 의존성 주입 전략)**:
  - 어떤 외부 연동망 API 접속 키(클라우드 호스팅 서브넷 주소, PostgreSQL 슈퍼 계정 패스워드 등)도 애플리케이션 하드코드나 Git 온라인 공개 저장소에 박혀 노출되지 않게 `python-dotenv` 패키지 분해 조치. 오직 배포 서버 내부 `.env` 파일과 Docker `environment` 파이프를 통해서만 런타임 시스템 메모리로 암묵 인젝션(Injection) 되도록 12-Factor App 보안 결계망 설정 완료.

## 10. 결론 : 운영 단계 데뷔 준비 궤도 달성 및 마스터플랜 청사진 (Conclusion & Horizon Plan)

단일 모놀리스 로컬 서버에서 버벅거리며 I/O 충돌을 겪던 프로젝트의 원시 태아기는 완전히 종식되었음. Docker 멀티 컨테이너 물리 격리와 GPU VRAM 비동기 파이프라인(Celery+Redis)의 블록들이 한 치의 오차 없이 맞물려 돌기 시작한 현재 아키텍처는, On-Premise 제한적 서버 장비(1 GPU) 조건 하에서도 WebRTC 화상 실시간성과 AI 추론 병행이라는 고난도 과제를 달성하며 수십 명의 동시 후보자 트래픽이 쏟아져도 거뜬히 버텨내는 고가용성 **실전 투입 가능 전단계(Production-Ready)**에 도달했음을 선언함.

**향후 시스템 초고도화(Hyper-Scale)를 향한 차세대 마스터플랜 로드맵은 다음과 같음:**

1. **LLM 모델의 초경량화/양자화(Quantization) 아키텍처 도입 (Inference Cost Optimization)**:
   - 현재 육중한 16-bit GGUF 파운데이션 모델의 GPU VRAM 점유 한계를 뚫기 위해, AWQ(Activation-aware Weight Quantization) 등 최신 4-bit 양자화 기법 및 PagedAttention 메커니즘을 지원하는 `vLLM` 초고속 전용 C++ 추론 백엔드로 엔진 전면 교체. 면접 세션 당 H/W 추론 단가와 응답 대기 지연(Latency)의 혁명적 압축 추진.
2. **Kubernetes(K8s) 기반의 무제한 분산 물리 스케일 아웃 (Autoscaling Orchestration)**:
   - 동시 면접 접속자가 급증할 시 단일 호스트 머신(Docker Compose)을 넘어, `interview_worker_gpu` 파드(Pod)들을 다수의 물리 이기종 서버 머신 노드 풀에 동적 복제(HPA) 분산 배치하는 오케스트레이션 구름 제어망 확보. (대량 트래픽 수용을 위해 메시지 브로커 이벤트 허브를 Redis에서 분산 파티셔닝에 특화된 `Apache Kafka` 스트림으로 마이그레이션 진단 검토).
3. **폴백(Fallback) 방어 기제의 AI 지능화 (RAG Retriever 확장 고도화)**:
   - 현재 LLM이 크래시 나면 정해진 딱딱한 정적 JSON 문장 세트만 내뱉는 재난 방어 로직을 진화시켜; DB(`pgvector`)에 수만 개의 모범 면접 세션 데이터를 아카이브하고 OOM 크래시가 터지는 즉시 거꾸로 CPU가 Vector Search L2 연산을 감행해 "현재 면접 문맥 흐름상 가장 유사할 법한 과거 다른 유저에게 출제했던 명대사/명질문 베스트"를 즉석에서 0.5초 만에 조립해 브라우저로 쏴주는 **초지능 백업 클라우드 브레인 플랜 (Zero-Downtime Smart Fallback)** 완전 구현.
4. **옵저버빌리티(Observability) 관제탑 센서망 통합 (APM Monitoring Center)**:
   - 프로덕트의 모든 숨겨진 핏줄(LLM 토큰 틱 생성 지연 구간, WebRTC UDP 패킷 로스율 빈도 파악, DB 트랜잭션 락 지연 수치)을 `LangSmith` 프롬프트 트레이싱 시스템은 물론, `Prometheus` 시계열 메트릭 파이프와 `Grafana` 중앙 대시보드 스택 기반 애플리케이션 모니터링 체계로 가시화하는 차세대 사이트 신뢰성 공학(SRE) 센터 건립에 돌입할 최적의 타이밍임.
