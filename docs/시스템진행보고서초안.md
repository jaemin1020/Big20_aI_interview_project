## 1. 프로젝트 총괄 개요 및 비전 (Project Overview & Vision)

### 1.1 프로젝트의 탄생 배경과 비전
Big20 AI 면접 프로젝트(A4 프로젝트)는 단순히 일방향적인 온라인 테스트 도구를 넘어서, 최전선 기술인 대형 언어 모델(LLM), 시각 인공지능(Vision AI), 음성 인식 및 합성(STT/TTS), 그리고 초저지연 양방향 통신망(WebRTC)을 한데 엮어낸 **실전형 차세대 면접 플랫폼**입니다.
본 시스템은 대형 클라우드 사업자의 매니지드 서비스(SaaS)에 종속되지 않고, 온프레미스(On-premise) 형태의 통제된 서버 네트워크 환경에서도 독립적이고 완벽한 B2B/B2C 면접 솔루션을 구동할 수 있도록 설계되었습니다. 

지원자의 답변 내용 자체의 논리성과 직무 적합성을 파악함은 물론, 발화 속도, 목소리 떨림, 시선 분산도, 미세 표정 변화 등 비언어적(Non-verbal) 태도까지 종합적으로 수집·분석하여, 사람 면접관보다 더 정량적이고 딥(deep)한 인사이트를 제공하는 종합 피드백 대시보드를 구축하는 것이 근본적인 비전입니다.

### 1.2 시스템 아키텍처 담당의 전략적 책무 및 핵심 라이브러리 스택
광범위한 기술 스택이 동원되는 만큼, 시스템 아키텍처 부서는 단순 배포 스크립트 작성을 넘어 다음과 같은 전방위적이고 거시적인 기술 경계를 총괄하며, 이 모든 과정을 **Docker 기반 가상화 환경**으로 완벽히 묶어냅니다.

1. **도커(Docker) 기반 인프라스트럭처 엔지니어링 (Infrastructure Engineering)**:
   - GPU 가속이 필수적인 컨테이너(LLM 추론 등)와 멀티코어 연산이 중심이 되는 컨테이너(파싱, WebRTC 스트리밍) 간의 자원 경합을 방지하기 위한 토폴로지 격리 설계.
   - 단일 `docker-compose.yml` 기반의 통합 오케스트레이션을 통한 완전한 시스템 이식성 확보. (NVIDIA Container Toolkit 연동을 통한 `deploy.reservations.devices` 설정 강제)
   - 격리된 내부망(VNet) 설정과 호스트 머신과의 포트 바인딩 정책(API 8000, WebRTC 8080/UDP 50000 대역), 네트워크 병목 제어 전략 수립.
   - **주요 기반 스택**: `Docker Engine`, `Docker Compose`, `NVIDIA CUDA Toolkit`
2. **애플리케이션 레이어 파이프라인 (Application Layer architecture)**:
   - 클라이언트의 HTTP REST 요청을 지연 없이 소화하기 위한 코어 API 게이트웨이 코디네이션.
   - 무거운 추론 작업을 백그라운드로 밀어내기 위한 분산 워커와 메시지 브로커 이벤트 기반(Event-Driven) 시스템 도입.
   - **주요 라이브러리**: `FastAPI` (고성능 비동기 API), `Celery` (비동기 Task 큐), `Redis` (메시지 브로커 및 단기 세션 캐싱)
3. **AI 미디어 파이프라인 및 서빙 (AI & Media Pipeline)**:
   - WebRTC 기반 멀티미디어 스트리밍 통신망과 비전 감정 분석.
   - RAG(Retrieval-Augmented Generation) 패러다임에 기반한 문서 청킹, 임베딩, 그리고 LLM 프롬프트 생성 체인 구조화.
   - **주요 라이브러리**: `aiortc` (WebRTC P2P 릴레이), `faster-whisper` (오프라인 고속 STT), `sentence-transformers` (텍스트 임베딩), `langchain` (프롬프트 제어), `MediaPipe` 및 `DeepFace` (시각 프레임 분석)
4. **데이터 아키텍처 및 영속성 (Data Architecture & Persistence)**:
   - 단순한 관계형 데이터 맵핑을 넘어 수만 차원의 벡터 배열을 DB 내에서 직접 유사도 탐색하게 해주는 통합 서치 구조 확립.
   - 실시간으로 쏟아지는 감정 점수와 대화 텍스트(STT) 로그를 트랜잭션 손실 없이 기록하기 위한 ERD 정규화 및 Indexing 전략 구성.
   - **주요 DB/드라이버**: `PostgreSQL 18`, `pgvector` (벡터 L2/Cosine 유사도 검색 확장 플러그인), `SQLModel` & `SQLAlchemy v2`, `psycopg3` (비동기 DB 드라이버)
5. **옵저버빌리티 및 보안 전략 (Observability & Security Defenses)**:
   - JWT를 활용한 무상태(Stateless) 인증 아키텍처와 사용자 Role(지원자, 관리자)에 따른 엄격한 RBAC 통제 계층 구현.
   - 각종 딥러닝 모델의 크리덴셜, 외부 API Key의 하드코딩 완전 배제(.env 주입) 및 시스템 런타임 로그 컨벤션 체계화.
   - **주요 시큐리티 모듈**: `PyJWT`, `passlib` (단방향 해시), `python-dotenv`

## 2. 초기 아키텍처 설계안 구상 및 한계점 대면 (Initial Design & Limitations)

프로젝트 기획이 막 시작되었던 단계에서는, 최우선 목표가 "PoC(개념 증명)를 위한 빠른 프로토타이핑"에 맞춰져 있었습니다. 이에 따라 아키텍처는 극단적으로 단순화된 **모놀리식 특성을 지닌 제한적 분산 구조**를 채택했습니다.

### 2.1 초기 인프라 및 AI 처리 설계안
- **모놀리식 단일 컨테이너 성향**: Frontend(React), Backend(FastAPI), Database(PostgreSQL), Media-Server를 한 묶음으로 올리면서, 백엔드 컨테이너 자체가 가벼운 AI 추론을 같이 담당하거나 단 1개의 범용 AI Worker만이 전체 AI 연산을 떠맡도록 구성했습니다.
- **동기식(Synchronous) LLM 호출**: Frontend에서 유저 발화가 끝난 이벤트를 전송하면, 백엔드는 즉시 외부LLM 혹은 로컬LLM 엔진에 질문 텍스트 생성을 요청(Request)하고, **해당 응답(Response)이 올 때까지 10~20초간 아무것도 하지 않고 쓰레드가 블로킹 대기**를 타는 전형적인 일대일 동기 호출망이었습니다.
- **Memory-based 상태 관리**: 면접 세션의 진행 상태(현재 몇 번째 질문인지, 사용자의 누적 대화 내역이 무엇인지)를 백엔드 서버 RAM 변수(Dict)에 의존하여 저장하고, 면접이 끝날 때만 DB에 덤프격으로 INSERT를 시도하는 방식이었습니다.

### 2.2 한계 부딪힘 현상 (Bottlenecks & Death Spirals)
본격적으로 STT 기반의 음성 인식이 추가되고, 브라우저가 캠 영상을 WebRTC로 끊임없이 밀어 넣기 시작하자 치명적인 구조적 결함이 노출되었습니다.
1. **자원 록(Resource Lock)에 의한 STT 병목**: STT가 오디오를 텍스트로 변환하는 동안, 동일한 컨테이너(또는 단일 워커)에 배정된 LLM 추론이 GPU 메모리를 순간적으로 고갈시키며 STT 프로세스가 OOM(Out of Memory)으로 강제 종료(Killed)되는 충돌이 잦았습니다.
2. **소켓 타임아웃(Socket Timeout)과 데드락**: 동기식 구조 탓에, AI 모델이 일시적으로 연산이 밀려 30초 이상 응답을 주지 않으면 Nginx나 로컬 웹서버 단말에서 Gateway Timeout(504)을 띄워버리며 면접 화면 전체가 하얗게 죽어버리는 데드락 아키텍처였습니다.

이러한 문제들은 확장이 절대적으로 불가능한(Un-scalable) 구조임을 방증했으며, 즉각적이고 대대적인 아키텍처 진화(Evolution)의 계기가 되었습니다.

## 3. 컴포넌트별 아키텍처 초정밀 설계 진화 과정 (Evolutionary Details)

병목 현상들을 격파하기 위해, 컴포넌트 간 결합도(Coupling)를 낮추고 응집도(Cohesion)를 높이는 **MSA(Microservices Architecture) 지향적 리팩토링**이 전방위적으로 추진되었습니다.

### 3.1 인프라스트럭처 레이어의 고도화 (Infrastructure Isolation)

#### 3.1.1 GPU/CPU 이기종 자원 격리 (Workload Segregation)
가장 시급했던 AI 연산 충돌을 막기 위해 Celery Worker 컨테이너를 물리적으로 두 개의 별동대로 분할했습니다.
- **`interview_worker_gpu` (`gpu_queue` 전담)**:
  - NVIDIA Container Toolkit 설정을 통해 호스트의 VRAM 가속을 단독으로 독점.
  - 임베딩 변환기(HuggingFace), 면접 꼬리질문 생성기, 세션 최종 종합 채점기 등 무거운 행렬 연산 기반 텐서 추론 태스크만을 구독.
- **`interview_worker_cpu` (`cpu_queue` 전담)**:
  - 다중 코어 CPU의 병렬 처리 능력을 극대화.
  - 업로드된 수십 장짜리 이력서(PDF) 텍스트/JSON 정밀 구조화 파싱 작업, 오디오 버퍼 변환, Whisper C++ 기반(Faster-whisper) 고속 음성 문자열 변환(STT) 등의 I/O 및 CPU 집약적 연산을 전담.
- **결과**: LLM이 아무리 최대 자원을 당겨 쓰며 텍스트를 만들고 있더라도, 백그라운드에서는 CPU 워커가 새로운 음성을 STT 변환하며 멈춤 없이 전진하는 완전한 비동기 평행 처리가 가능해졌습니다.

#### 3.1.2 WebRTC 네트워크 토폴로지 교정
- 화상 면접의 심장인 `aiortc` 미디어 서버는 UDP 기반의 ICE(Interactive Connectivity Establishment) 교섭을 반드시 필요로 합니다. 
- 단일 Bridge 네트워크 내부에 갇힌 도커 구조의 한계로 Candidate Gathering이 실패하는 현상을 타파하기 위해, `docker-compose.yml` 상에 P2P 통신용 미디어 패킷 포트 대역(예: `50000-50050/udp`)을 명시하여 호스트OS 포트와 매핑 개방 시켰습니다. 이를 통해 TURN 서버 없이도 로컬 및 내부망에서 딜레이 제로에 수렴하는 비디오/오디오 스트리밍 길을 개통했습니다.

#### 3.1.3 무중단 상태 자동화 DB Init
- PostgreSQL이 올라오는 즉시 인프라 레벨의 Init 스크립트가 돌아가도록 설계. ORM `db_models`를 스캔하여 미생성된 스키마 테이블을 안전하게 Create하고, 시스템에 필수적인 마스터 채용자(`admin`/`recruiter`) 계정과 가상의 IT/경영/기획별 표준 회사 질문 룰셋(Rule Set) 레코드를 자동 주입(Seeding)하게 하여, 컨테이너를 재배포(Down->Up) 할 때마다 데이터가 날아간 무의미한 환경이 되는 것을 방어했습니다.

### 3.2 애플리케이션 및 비동기 이벤트 레이어 (Async Event-Driven Core)

#### 3.2.1 동기적 블로킹 파괴와 Redis 메시지 큐 장착
응답 지연으로 인한 서버 사망 문제를 근절하기 위해, 백엔드의 동작 원리를 철저히 Event-Driven으로 재설계했습니다.
- 클라이언트(React)가 "다음 질문 주세요"라고 API(`GET /next_question`)를 찔렀을 때, 
- 백엔드(FastAPI)는 직접 LLM에 추론을 돌리지 않습니다. 즉시 `Redis` 메시지 브로커로 `task_generate_question` 이라는 꼬리표를 붙인 페이로드(세션 ID, 면접관 정보)를 가볍게 밀어 넣고 0.1초 만에 클라이언트에게 "작업 들어갔음(202 Accepted)"을 리턴해버립니다.
- 프론트엔드는 블로킹 되지 않은 쾌적한 화면을 유지하며 주기적으로 폴링(Polling)하거나 WebSocket을 열어두고 결과가 떨어지기를 느긋하게 기다리게 됩니다. 이로써 504 Gateway Timeout의 악몽에서 영구 탈출했습니다.

#### 3.2.2 무상태성(Stateless) 아키텍처와 분산 메모리
메모리가 죽으면 면접 진행률도 날아갔던 구시대적 방식에서 벗어나,
모든 세션 데이터(현재 1번 문항 진행중인가, 유저가 방금 한 말이 무엇인가 등)를 PostgreSQL에 100% 트랜잭션 기록(`status: LIVE`, `transcript` 등)시켜 영속화했습니다.
또한 토큰 리프레쉬나 WebRTC Signalling 연결 객체(peer connection) 같이 DB에 쓰기엔 무겁고 너무 잦은 단기 데이터 캐시들은 `Redis` 의 in-memory key-value 쌍에 TTL(Time To Live)을 걸어 저장함으로써 로드 속도 한계를 극복했습니다.

### 3.3 데이터 아키텍처 및 융합 검색엔진망 (PostgreSQL RAG)

#### 3.3.1 RDBMS와 Vector 서치의 융합: `pgvector` 의 이식
본 시스템은 "지원자의 이력서", "기업의 인재상", "기존 출제되었던 꼬리질문" 세 가지 데이터를 복합 참조(RAG)하여 궁극의 맞춤형 프롬프트를 만듭니다.
이때 외부 라이브러리인 검색 엔진(Milvus, Pinecone 등)을 도입하지 않고, 시스템 복잡도를 극소화하면서 단일 연결점(Single Point of Truth)을 확보하기 위해 PostgreSQL 18 인스턴스에 `pgvector` Extension을 탑재했습니다.
- **RAG 파이프라인 효과**: 지원자가 이력서 PDF를 올리면 -> CPU 워커가 텍스트 청킹(Chunking)을 하고 -> GPU 워커가 이를 HuggingFace 로컬 모델로 임베딩하여 1024차원의 `[0.012, -0.344, ...]` 식의 벡터 리스트를 뽑아냅니다.
- **Hybrid Querying 기법**: 이것을 DB의 `resumes.embedding` 컬럼으로 직행 저장합니다. 이후 면접관(LLM)이 질문을 구성할 때, 단순 SQL 쿼리문안에 `ORDER BY resume.embedding <-> 질문벡터` 구문(L2 Distance 연산)을 날려 관계형 데이터 권한(내 이력서만)과 의미론적 유사성 결과를 일격에 Join하여 반환받는 압도적 효율을 창출했습니다.

#### 3.3.2 데이터 정규화 및 ERD 인덱스 고도화 
- 트랜스포머/LLM 시스템 구조에 맞춰 Backend의 `SQLModel` 코드를 수차례 리팩토링했습니다. 
- 특히 가장 누적이 잦은 `Transcripts`(대화 기록) 로우가 어떠한 질문(`question_id`)의 답변인지를 명확히 외래키(FK) 트래킹할 수 있도록 연관 매핑을 단단히 했으며.
- `Emotions` 같이 초당 수차례 쏟아지는 감정 분석 지표들은 별도 테이블로 분리해 Join 과부하를 일으키는 대신, Transcripts 테이블과 Interviews 테이블 내부의 `JSONB` 컬럼(`emotion_summary` 등)으로 원자화 편입시켜 I/O 레이턴시를 제로에 가깝게 소화해냈습니다.

### 3.4 AI 모델 최적 서빙 및 미디어 파이프라인 (AI Serving & RTSP)

#### 3.4.1 Client-side 지향적 STT / TTS 파이프라인 혁신
구조상 미디어 서버단에서 영상을 파이프하는 중간에 FFmpeg등으로 오디오를 쪼개서 서버 로컬 Whisper로 넘기는 방식(초기 구상)은 도커 내부 소켓 연결을 폭파시키는 원흉이었습니다.
이를 **Client-side Audio Buffering Architecture**로 역발상 전환했습니다.
- 프론트(React 브라우저)에서 `AudioWorkletNode`를 사용하여 백그라운드 워커 스레드로 PCM 고음질 마이크 스트림을 분리 수집합니다. 메인 UI 스레드(Rendering)는 절대 방해받지 않습니다.
- 프론트가 화상(Video)은 MediaServer(WebRTC)로 던지지만, 끊어진 음성 청크 덩어리는 비동기 API를 거쳐 즉시 최적화된 STT 원격 워커(Deepgram/Whisper)로 쏘아 올립니다. 이 분산 구조를 통해 말하는 즉시(1~2초 내) 화면에 자막(Transcript)이 박히는 무서운 리얼타임을 이끌어냈습니다.

#### 3.4.2 Vision 연산과 백그라운드 폴링 구조
- AI Media-Server에서는 들어오는 비디오 패킷에서 파이썬 `ndarray`로 프레임을 훔쳐(Overriding Protocol) MediaPipe Face Landmark 에 넘깁니다. 
- 이때, 모든 30 FPS 프레임을 모델에 통과시키면 CPU 100% 코어 점유 크래시가 발생하므로, 타임 델타 로직을 적용한 **초당 5프레임 샘플링(Sampling) 제한 방어망**을 완벽 구현했습니다. 그 결과 서버 응답 지연을 방어하면서도 "긍정 40%, 무표정 60%" 등의 통계적 유의성을 잃지 않는 감정 스코어링의 대동맥을 열었습니다.

## 4. 시스템 아키텍트(jaemin1020)의 독보적 개발 성과 및 기술적 기여 (Architect Contributions)

단순한 코드 한두 줄 기여가 아닌, 전체 아키텍처의 철학을 지배하고 각 병목 구간의 코어를 집도(Surgery)한 시스템 아키텍트 `jaemin1020` 계정의 방대한 기술 설계 및 개입 내역입니다.

### 4.1. 베이스 인프라 및 모노레포(Monorepo) 생태계 정립
본 시스템은 서로 다른 프레임워크와 구동 환경이 복잡하게 얽혀 있었습니다.
- 이를 타파하기 위해 최상위 루트 디렉토리를 기준으로 `frontend`, `backend-core`, `ai-worker`, `media-server`의 모노레포 구조를 최초 생성하고 각 디렉토리의 진입점(Entry)을 통일했습니다. 
- 복잡한 도커 명령어 조합 없이 단순 `docker-compose up` 하나만으로 수 개의 이기종 네트워크 컨테이너가 약속된 포트 매핑하에 일제히 기동 될 수 있도록 스크립트 기반을 완전 구상 및 이식했습니다.
- Git 트래킹 시 불필요하게 딸려 올라가는 GPU 가중치 파일(GGUF, Tensor 등), 패키지 캐시(`__pycache__`, `node_modules`), 개인 API 키가 들어있는 `.env` 파일들을 루트 및 모듈별 `.gitignore` 필터링 규칙으로 섬세하게 직조해 형상 관리 자원을 초경량화했습니다.

### 4.2. 코어 비즈니스 지형도 파괴적 혁신 및 AI/데이터 결합
- **로컬 LLM 크래싱 방지용 Fallback (안전장치) 시스템 구현**:
  - LLM 노드가 가용 VRAM 한계를 뛰어넘어 무한 추론 Loop나 메모리 에러를 뿜으며 타임아웃(Timeout) 큐에 처박힐 때, 서비스 장애 화면을 띄우는 참사를 예방했습니다. 
  - Celery Exception이나 재시도(Retry) 허용 횟수(`max_retries=3` 등)를 초과하게 될 경우, 코드 흐름을 꺾어 **DB에서 하드코딩된 사전 예비 질문(Fallback Set)을 강제로 추출해 반환**시키는 방어 트랜잭션 로직을 아키텍트 주관하에 이식, 100% 면접 진행 무중단이라는 기적을 설계했습니다.
- **RAG / Vector DB 스키마 100% 동기화 작업**: 
  - 백엔드 레이어의 `db_models.py`와 `database.py` 안에 혼재되어 있던 복잡한 1:N 쿼리 연결, 그리고 LLM이 요구하는 1024 차원의 PGVector 선언문을 손끝으로 하나하나 재검수하여, ERD 구조와 물리적 SQL 테이블이 완벽성을 이루도록 통합 튜닝을 실행했습니다.

### 4.3. 방대하고 집요한 대형 TDD / 기술 명세서(Technical Docs) 자산화 맹활약
개발된 코드 조각들이 휘발되지 않도록, 전체 시스템의 작동원리와 구조를 파헤치는 대규모 문서화 작업(Technical Writing)을 프로젝트 처음부터 끝까지 총괄 지휘했습니다.
- **시스템 아키텍처 디자인 명세서(SAD) 전면 고도화**:
  - 단순 서술 수준이던 문서를, C4 모델 정적 구조 분해(Frontend~AI Worker 계층별 역할 매핑), 런타임 동적 시스템 워크플로우(WebRTC 미디어 패킷 핸드셰이킹부터 LLM 추론까지 이어지는 완전 무결한 핑퐁 프로세스) 시퀀스 분해 등 초정밀 머메이드(Mermaid) 다이어그램으로 시각화 및 문서화 배포 완료.
- **문법 포대갈이 및 구버전 호환 렌더링 튜닝**:
  - 사내망 또는 깃허브 등 구버전 환경(Mermaid 8.8 이하) 등에서도 복잡한 ERD 명세와 액터 노드들이 에러 없이 미려하게 렌더링 될 수 있게, 문법 디버깅과 다이어그램 재포맷팅 작업을 기를 쓰고 완수했습니다. 
  - 그 밖의 모든 보안, DB 연동 가이드, 트러블슈팅, 최적화 레거시 리포트 등 수십여 개의 마크다운 지식 베이스가 아키텍트의 손끝에서 묶여, 사실상 프로젝트 생태계의 교과서(Bible) 인프라로 안착했습니다.

## 6. 치열한 기술 병목 극복 및 트러블슈팅 열전 (Critical Troubleshooting Portfolio)

실제 프로덕션 서버를 빌딩하면서 발생한 10가지 이상의 대형 참사와 이를 엔지니어링 솜씨로 무력화시킨 여정입니다. 문서 용량 제약상 핵심 사건만 상세 발췌합니다.

### 6.1 치명적 ModuleNotFoundError: Celery 네임스페이스 격리 실패 사태
- **증상 관찰**: Backend 컨테이너가 면접 꼬리 질문을 발행하기 위해 `@celery_app.task` 로 등록된 워커의 함수를 직접 `import tasks.llm_generator` 문법으로 호출하는 순간 컨테이너 크래시 터짐.
- **원인 기전 분해**: 도커 상 물리적으로 백엔드와 AI 워커는 "분리된 컨테이너 공간과 분리된 파일시스템"을 갖기 때문에, 물리 모듈을 `import`하는 것은 분산 처리 MSA 사상에 정면 위배되는 행위.
- **설계적 처방**: "Signature" 패턴 기반 위임으로 아키텍처 리팩토링 수행. Backend 코어에서는 어떠한 워커 쪽 스크립트도 직접 import 하지 않게 소스를 갈아엎고, `celery_app.send_task("태스크_원격_스트링_명칭", args=[...])` 의 매직 스트링 포맷으로 오직 Redis 메시지 브로커 안에 JSON 명세표만 쑤셔 넣는 **"완전한 객체 지향적 원격 프로시저 호출(RPC) 디커플링"**을 이룩함.

### 6.2 오디오/비디오 메인 스레드 점유로 인한 UI 멈춤 현상 발생건
- **증상 관찰**: React 프론트엔드가 면접장에 들어가면, 노트북 팬(Fan)이 굉음을 내면서 마우스 스크롤이 끊기고 애니메이션이 뚝뚝 끊기는 치명적 렉(Lagging) 유발됨.
- **원인 기전 분해**: `getUserMedia` 로 얻어온 오디오 마이크 PCM 버퍼 스트림을 `ScriptProcessorNode`의 EventListener 기반 자바스크립트 "메인 렌더 스레드" 에서 콜백 형태로 배열을 썰고 자르고 있었음.
- **설계적 처방**: 메인 스레드의 부하를 덜어내기 위해 브라우저 엔진 백그라운드로 도피하는 최신 모던 API **`AudioWorkletProcessor`** 구조판으로 웹 오디오 코어 엔진 모듈 마이그레이션 단행. 결과 브라우저 메인스레드 점유율을 50% 밑으로 떨어뜨리며 60프레임급 쾌적한 반응형 면접 프론트 완성.

### 6.3 볼륨 바인딩 권한 충돌과 이력서 파싱 증발 파동 
- **증상 관찰**: 사용자가 파일을 올리고 Backend가 분명히 응답(200 OK)을 쏴줬는데, 30초 뒤 DB를 열어보면 이력서 파싱 상태가 통째로 `failed`거나 None으로 비어있음.
- **원인 기전 분해**: Backend 컨테이너 단의 `/app/uploads` 폴더 공간과 CPU Worker 컨테이너 파일 시스템 공간이 완전히 남남으로 독립되어 있어 파일이 유실된 것처럼 접근 액세스가 차단되었던 것.
- **설계적 처방**: `docker-compose.yml` 볼륨 명세단에 `volume:` 공유 설정 네임스페이스를 글로벌로 잡아두고, 양쪽 컨테이너의 Host 경로를 하나의 동일한 가상 공유 폴더로 관통 연결시키는 브릿지 탑재함으로써 파일 IO 문제를 말끔히 무중단 연결.

## 7. 시스템 스펙 및 성능 검증 벤치마킹 분석 리포트 (Performance & Benchmark)

자원 최적화 설계의 과실을 숫자로 대변하는 핵심 퍼포먼스 체크포인트 요약.

- **미디어 스트리밍 실시간성(RTSP)**: 
  - Host OS 포트바인딩 기반 UDP 홀펀칭 트래버스 적용 이후, 네트워크 지터(Jitter)로 영상/음성이 심하게 밀리던 현상 제압. 
  - 평균 화상 전송 RTT(Round Trip Time) 왕복 500ms 미만의 극도의 로우 레이턴시 달성으로 대화 핑퐁 장애 제거.
- **대규모 추론 (EXAONE-3.5) 병렬 속도**:
  - Task 큐잉 딜레이 및 GPU(VRAM) 텐서 로딩, 그리고 RAG 유사도 검색까지 풀 스코어링 소요.
  - 지원자의 답변 텍스트(STT 결괏값)가 꽂힌 순간부터 완전히 새로운 날카로운 꼬리질문 텍스트가 DB를 타고 소켓으로 터져나올 때까지의 엔드-투-엔드(End-To-End) 평균 생성 리드타임을 10초 내외 안전권으로 끊어냄. (타임아웃은 전혀 관측되지 않음)
- **STT (Faster-Whisper 기반) 청크 생성 체력**:
  - 클라이언트에서 보내오는 5초/10초 단위의 정적 감지(VAD) 오디오 웨이브 조각들을 변환하는데 걸리는 추론 시간 단 1~2초대. "거의 사용자가 말을 멈추고 숨 돌리기 무섭게 텍스트로 박히는" 수준의 경이로운 멀티 프로세스 스루풋 입증 완료.

## 9. 프레임워크 수준의 보안 하드닝 및 컴플라이언스 관리 수준 (Security Stance)

- **JWT 토큰 암호화 체계(Stateless)**: 서버 메모리에 세션 상태를 전혀 남기지 않는 완전한 RESTful 원형 유지. 시크릿 해시 함수(HS256) 기반 토큰 위변조 원천 파괴 방어벽 구축. `OAuth2PasswordBearer` 스킴 통합하여 FastAPI Depends 체인 통한 악성 유저 차단망 완비.
- **엔드포인트별 치밀한 권한 분리 통제방 (RBAC - Role Based Access Control)**:
  - `admin`, `recruiter`, `candidate` 3계층 유저 팩션을 정의.
  - 내 이력서 파싱 결과를 몰래 남의 계정으로 GET 하려는 접근을 도메인 모델 수준에서 `if current_user.id != request.user_id` 방어 트리거로 튕겨버려 엄밀한 멀티 테넌트 무결성을 수호.
- **운영 체제 환경 은폐화 (.env 의존성 주입 전략)**:
  - 어떤 외부 API 통신 열쇠(딥그램 API, 클라우드 호스팅 서브넷 주소, DB 슈퍼 계정 패스워드)도 앱 코드나 깃헙(GitHub) 온라인 공개 저장소에 박혀 올라가지 못하게 분해 조치. 오직 배포 서버 내부 `.env` 파이프를 통해 시스템 메모리로만 암묵 인젝션(Injection) 되도록 봉인 결계 설정 완료.

## 10. 결론 : 운영 단계 데뷔 준비 궤도 달성 및 마스터플랜 청사진 (Conclusion & Master Plan)

단일 모놀리스 로컬 서버에서 버벅이던 프로젝트의 원시 태아기는 완전히 종식되었습니다. 도커 컨테이너 격리와 GPU VRAM 비동기 파이프라인의 조각들이 맞물려 돌기 시작한 현재 구조는, On-Premise 제한적 서버 장비 상황 하에서도 수십 명의 후보자 트래픽이 쏟아져도 거뜬히 버텨내는 고가용성 **실전 투입 가능 전단계(Production-Ready)**에 도달했음을 선고합니다.

**향후 시스템 초고도화를 향한 차세대 마스터플랜 로드맵은 다음과 같습니다:**

1. **LLM 모델의 초경량화/양자화(Quantization) 아키텍처 도입**:
   - 현재 육중한 GGUF 파운데이션 모델의 GPU VRAM 점유 한계를 뚫기 위해, AWQ 등 최적화된 4-bit 양자화 및 vLLM 초고속 추론 백엔드로 교체. 세션 당 추론 단가와 응답 지연의 혁명적 압축 추진.
2. **Docker Swarm(또는 K8s) 기반의 분산 물리 스케일 아웃**:
   - 동접자가 급증할 시 단일 노드 호스트를 넘어, 워커 노드들을 서로 다른 물리 서버에 증식(Scale-Out)시켜 컨테이너를 복제 분산 배치하는 오케스트레이션 클러스터 레벨 통치권 확보 추진. (메인 브로커는 Redis에서 대용량 분산 처리에 특화된 Kafka 이벤트 스트림 허브로 교체 검토).
3. **폴백(Fallback) 방어 기제의 AI 지능화 (Retriever 확장)**:
   - 현재 LLM 마비시 정해진 딱딱한 문장만 내뱉는 로직을 진화시켜; DB(pgvector)에 축적된 수만 개의 모범 면접 세션 데이터를 거꾸로 Vector Search 연산해 "현재 문맥상 가장 유사할 법한 과거 명대사/질문"을 즉석에서 Retriever가 조립해 쏴주는 **초지능 백업 클라우드 브레인** 기능 구현.
4. **옵저버빌리티 관제탑 통합 (APM Monitoring Center)**:
   - 프로덕트의 모든 맥박(LLM 지연, 로그 크래시 빈도, DB 트랜잭션 수치)을 LangSmith 트레이싱 시스템과 더불어 Prometheus / Grafana 대시보드 스택 기반 C-레벨 요약 관리 창으로 구조화하는 사이트 신뢰성 공학(SRE) 기반 구축에 돌입할 예정입니다.
