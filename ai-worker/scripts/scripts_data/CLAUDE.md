# 면접 답변 평가 프로젝트 - 실행 지침

## 1. 프로젝트 개요

- **대상**: 12명의 참여자, 인당 약 510개 기술 면접 답변 (IT 전반 - 네트워크, OS, DB, 자료구조, 알고리즘, 소프트웨어 공학, 딥러닝/ML/AI 등)
- **평가자 역할**: IT 전반 분야 전문가로서 사실 기반의 엄격한 평가 수행
- **핵심 원칙**: 거짓 없이, 틀린 것은 틀렸다고 명확히 지적

## 2. 평가 순서 및 방식

### 순서
```
01_준호 → 02_린 → 03_재민 → 04_여림 → 05_지훈 → 06_승재
→ 07_하량 → 08_다윗 → 09_명혁 → 10_윤재 → 11_시현 → 12_유라
```

### 방식
- **한 명씩 순차 처리** (병렬 처리 금지, 품질 우선)
- **chunk 단위 처리**: 1인당 11개 분할 파일(각 50개)을 순서대로 읽고 평가
- 각 chunk 파일은 2회에 나눠 읽기: 1회차 offset=1,limit=100 (앞 25개) / 2회차 offset=100 (뒤 25개)
- 한 chunk 읽기 → 평가 → **중간 메모 파일에 누적 저장** (`eval/{번호}_{이름}_메모.md`)
- 11개 chunk 완료 후 → 메모 파일 기반으로 최종 평가 MD 작성 (`eval/{번호}_{이름}_평가.md`)
- **매 1명 완료 후 사용자에게 결과를 보여주고 다음 진행 여부 확인**
- 메모 파일은 최종 평가 MD 완성 후에도 **보존** (답변 수정 시 chunk별 상세 평가 참조용)

### 데이터 파일 구조
- **원본**: `json/` 디렉토리 (12개 JSON, 수정 금지)
- **분할본**: `json/split/` 디렉토리 (132개 파일, 50개 단위)
  - 명명 규칙: `{번호}_{이름}_{chunk번호}.json` (예: `01_준호_001.json`)
  - 1인당 11개 파일: `_001.json`~`_010.json`(각 50개) + `_011.json`(나머지 4~10개)
  - 참여자별 질문 수: 준호~시현 각 510개, 지훈 509개, 유라 504개 (총 6,113개)
- **주의**: `01_준호` 분할 파일은 답변 내 LaTeX 수식(`\gamma`, `\log` 등)과 이스케이프 안 된 따옴표로 인해 JSON 파서로 읽히지 않음 → Read 도구로 텍스트 읽기 사용

## 3. 평가 기준 (prompt.md 기반)

### 3-1. 필수 통과 조건 (하나라도 위반 시 FAIL)
| # | FAIL 조건 |
|---|-----------|
| F1 | 질문의 핵심 의도를 정확히 다루지 못한 경우 |
| F2 | 첫 문장에서 문제/판단 대상이 명확히 정의되지 않은 경우 |
| F3 | 기술 질문에 맞는 기본 구조를 갖추지 못한 경우 |
| F4 | **사실 오류 또는 검증이 어려운 주장이 포함된 경우** |
| F5 | 답변이 4문장 이하인 경우 |

### 3-2. 평가 항목별 등급 (S / A / B / C / D)
| 항목 | 평가 내용 | 등급 기준 |
|------|-----------|-----------|
| A. 질문 적합성 | 질문 핵심에 정확히 답했는가 | S: 완벽 적합 / A: 적합 / B: 약간 벗어남 / C: 일반론 경향 / D: 핵심 이탈 |
| B. 구조 적합성 | 문제정의→원리→제약 흐름이 있는가 | S: 모범적 / A: 자연스러움 / B: 구조 있으나 불완전 / C: 구조 미흡 / D: 무구조 |
| C. 논리 완성도 | 문장 간 인과관계가 명확한가 | S: 철저한 논리 / A: 명확 / B: 부분적 비약 / C: 논리 허점 다수 / D: 비논리적 |
| D. 기술적 정확성 | 개념/용어가 사실적으로 정확한가 | S: 완벽 정확 / A: 정확 / B: 사소한 부정확 / C: 오류 포함 / D: 심각한 오류 |
| E. 현실성/제약 인식 | 비용/성능/운영 제약을 고려했는가 | S: 깊은 통찰 / A: 적절히 언급 / B: 피상적 언급 / C: 거의 없음 / D: 전혀 없음 |
| F. 표현/형식 | 면접 톤, 불필요한 수식어 없는가 | S: 모범적 / A: 적절 / B: 약간 과장 / C: 나열식/추상적 / D: 형식 미달 |

## 4. 전문가 관점의 기술 검증 원칙

평가자는 IT 전반(네트워크, OS, DB, 자료구조, 알고리즘, 소프트웨어 공학, 딥러닝/ML/AI 등) 전문가로서 다음을 엄격히 검증한다:

1. **용어 정확성**: 기술 용어의 정의가 학술적으로 정확한가
2. **인과관계**: "A이므로 B이다"의 논리가 기술적으로 타당한가
3. **과장/단정 여부**: 특정 기법을 유일한 정답처럼 단정하지 않았는가
4. **최신성**: 더 이상 유효하지 않은 주장이나 outdated된 정보가 없는가
5. **깊이**: 표면적 설명에 그치지 않고 "왜"에 대한 답이 있는가
6. **오해 유발**: 정확하지만 오해를 유발할 수 있는 표현이 있는가

## 10. 답변 수정 작업 (Phase 2)

### 10-1. 목표
- 12명 전원의 종합 등급을 **A 이상**으로 개선
- FAIL 답변 1,534개 확장 + 기술 오류 43건 교정 + 오탈자 전수 교정

### 10-2. 수정 원칙
- **원본 보존**: `json/`, `json/split/` 디렉토리의 원본 파일은 절대 수정하지 않음
- **별도 수정본**: `json_fixed/` 디렉토리에 수정된 JSON 저장
- **기존 내용 기반 확장**: 기존 3~4문장의 핵심을 유지하며 5~6문장으로 자연스럽게 확장 (트레이드오프, 실무 사례, 제약 조건 등 보강)
- **전부 수정**: FAIL 확장 + 기술 오류 교정 + 오탈자 교정을 함께 처리
- **참여자 스타일 유지**: 각 참여자의 답변 톤과 표현 패턴을 유지하되, 과도한 반복 패턴은 다양화

### 10-6. 품질 검사 (참여자별 완성 후 2회)

1인분의 전 chunk 수정 완료 후, **2회의 품질 검사**를 실시한다:

#### 1차 품질 검사 (자동 검증)
- `json_fixed/` 수정본을 chunk 단위로 다시 읽기
- 수정된 FAIL 답변이 5문장 이상인지 확인
- 기술 오류가 올바르게 교정되었는지 확인
- 오탈자가 잔존하지 않는지 확인
- 새로운 기술 오류가 유입되지 않았는지 검증
- 문제 발견 시 즉시 수정 반영

#### 2차 품질 검사 (평가 기준 재적용)
- 수정된 답변에 대해 평가 기준(섹션 3)을 재적용
- FAIL 조건(F1~F5) 전수 재검사
- 항목별 등급(A~F) 재평가하여 A 이상 달성 여부 확인
- 미달 답변이 있으면 추가 수정 후 재검사
- 통과 시 해당 참여자의 수정 완료로 확정

#### 품질 검사 결과 기록
- 검사 결과를 `eval/{번호}_{이름}_검수.md`에 기록
- 1차/2차 검사 통과 여부, 추가 수정 사항, 최종 등급 기록
